# Incident runbook: lifecycle observability, failures, and rollback

## Scope

This runbook covers incident response for:

- Agent analysis request failures and latency regressions
- Trade execution failures and risk-rule rejections
- Authentication anomalies (invalid signatures, malformed/expired sessions, repeated login failures)
- Alert bursts generated by lifecycle observability

Primary instrumentation is implemented in:

- `lib/observability/lifecycle.ts`
- `app/api/agent/route.ts`
- `app/api/execution/route.ts`
- `app/api/auth/login/route.ts`
- `lib/auth.ts`

## Signals and what they mean

### Correlation IDs

- Every request should include/emit:
  - `x-correlation-id`
  - `x-request-id`
- Use these IDs to trace one request across:
  - inbound API request
  - agent orchestration (including sub-agent metadata on coordinator flow)
  - execution/rejection result

### Structured logs

All lifecycle logs are JSON with:

- `component=agent-lifecycle`
- `severity` (`debug` | `info` | `warn` | `error`)
- `event`
- contextual fields (`correlation_id`, `request_id`, `owner_user_id`, `agent_id`, etc.)

Important events:

- Analysis:
  - `analysis_request_received`
  - `analysis_request_completed`
  - `analysis_request_failed`
  - `analysis_response_schema_validation_failed`
- Execution:
  - `execution_request_received`
  - `execution_request_completed`
  - `execution_request_rejected`
  - `execution_request_failed`
  - `execution_failure_burst_alert`
- Auth:
  - `login_success`
  - `login_failed`
  - `auth_anomaly`
  - `auth_anomaly_burst_alert`

### Metrics

Tracked in-memory in lifecycle observability:

- Analysis latency:
  - `total`
  - `failures`
  - `avgLatencyMs`
  - `maxLatencyMs`
- Execution outcomes:
  - `success`
  - `failure`
  - `consecutiveFailures`
  - `rejectionReasons` (reason -> count)
- Auth anomalies:
  - `anomalies`
  - `anomaliesInWindow`
  - `windowMs`
- Alerts:
  - `executionFailureBursts`
  - `authAnomalyBursts`
  - last alert timestamps

### Alert thresholds

Tune via env vars:

- `EXECUTION_FAILURE_ALERT_THRESHOLD` (default `3`, min `2`)
- `AUTH_ANOMALY_ALERT_THRESHOLD` (default `5`, min `3`)
- `AUTH_ANOMALY_WINDOW_MS` (default `600000` ms, min `60000`)

## Triage procedure

1. **Open logs for the incident window** and filter by `component=agent-lifecycle`.
2. **Locate alert event(s)** first (`*_burst_alert`) and extract `correlation_id`/`request_id` where available.
3. **Pivot by correlation ID** to reconstruct request lifecycle.
4. **Classify incident type**:
   - analysis latency/failures
   - execution rejection/failure burst
   - auth anomaly burst
5. **Determine blast radius**:
   - count unique `owner_user_id` impacted
   - list impacted pairs/agents/signals
6. **Stabilize** (see immediate mitigations below).
7. **Verify recovery** by confirming alert rate drops and success events recover.

## Immediate mitigations

### Repeated execution failures

- If failures are dominated by one `rejection_reason`, validate risk settings for impacted users.
- If failures are external/provider failures, temporarily pause automated scheduling for impacted accounts.
- Reduce traffic by disabling/reducing scheduler triggers while investigating downstream dependency health.

### Auth anomaly bursts

- Check for spike in `SESSION_SIGNATURE_MISMATCH` / malformed token reasons.
- Rotate `AUTH_SECRET` only if compromise is suspected (forces re-auth for all active sessions).
- Temporarily throttle or block abusive IP/user-agent patterns at edge/network layer.

### Analysis latency regressions

- Confirm if slowdown is isolated to a specific agent ID or coordinator flow.
- Validate provider-side latency and recent deployment changes.
- If needed, switch to degraded mode (manual analysis, reduced scheduler activity) until latency normalizes.

## Rollback procedures

> Use staged rollback: config rollback first, then code rollback only if needed.

### 1) Configuration rollback (preferred)

1. Revert alert thresholds to previous known-good values.
2. Revert recent provider/config toggles affecting agent/execution flow.
3. Redeploy config-only change.
4. Confirm alert and failure rates normalize.

### 2) Application rollback

1. Identify last known-good commit SHA.
2. Deploy rollback release to staging first (if available), then production.
3. Run smoke tests:
   - login
   - `/api/agent` happy path
   - `/api/execution` happy path and one expected rejection path
4. Confirm structured logs still present and correlation IDs returned in headers.

### 3) Data rollback (only when required)

- For user/session/auth data incidents, restore from latest consistent snapshot only if data corruption occurred.
- Coordinate with database owner; validate RPO/RTO and communicate expected user impact.

## Post-incident checklist

1. Record exact alert trigger, timeline, and root cause.
2. Add or tune rejection reason categorization if too many fall into generic buckets.
3. Adjust thresholds/windows if alerting proved too noisy or too slow.
4. Add a regression test for the failed path.
5. Update this runbook with newly observed failure mode.
